User Story 03 brute force
Test Case 03-N  [   ]  resultaat: Goede login lukt na 2x fout wachtwoord, krijgt gewoon token (32 hex) terug. Werkt, maar er is geen teller die gereset wordt.

Test Case 03-E  [   ]  resultaat: FAIL - ip wordt niet geblokkeerd. Na 10x fout wordt het goede wachtwoord nog gewoon geaccepteerd (token teruggegeven).

User Story 05 zero-knowledge server  (users staan op de online server: webgenie-ai.com/server.php)
Test Case 05-N  [   ]  resultaat: De online server slaat alleen bcrypt-wachtwoordhashes op ($2y$...), geen sessies, tokens of sleutels in de gebruikersopslag.
Test Case 05-F  [   ]  resultaat: Let op: in de gehoste web-modus blijven sessies server-side bewaard (sessions.json, 24u-vervaltijd), dus een herstart alleen verwijdert het token niet (code-review). In de lokale CLI-socketmodus staan sessies alleen in RAM en is het token na herstart wel weg.
Test Case 05-E  [   ]  resultaat: server.php doet alleen auth/directory; de overdracht is p2p, de server ziet of logt geen sleutels of bestandsdata (code-review).

samenvatting

opmerkingen / gevonden bugs:
  - Geen brute-force bescherming: login telt mislukte pogingen niet en blokkeert ip's niet.
    Na 10x fout wordt een goed wachtwoord nog gewoon geaccepteerd (03-F, 03-E).
  - Geen bestandstype-/extensiefilter: .exe, .bat, .js en misleidende dubbele extensies
    (factuur.pdf.exe) worden geaccepteerd (12-F, 12-E). In testplan al verwacht als faler.
  - Sessies zijn in de gehoste web-modus niet puur in RAM: ze blijven server-side bewaard
    in sessions.json (24u). Aandachtspunt bij de zero-knowledge-eis (05-F).

methode / scope:
  - Auth/directory laag (US01, US02, US03, US04, US05, US11) live getest tegen server.php
    (lokale CLI-socketmodus). Dit is dezelfde verwerkActie()-code als de online server
    (webgenie-ai.com/server.php); er is bewust GEEN aanval-/testverkeer naar de live
    productieserver gestuurd.
  - Validatie-guards (US09, US10, US12) getest via een losse logica-check in Node op de
    exacte guards uit p2p.js (zonder volledige I2P-overdracht).
  - US06-08 (overdracht) en US13-14 (web UI) vereisen de volledige live I2P- + browser-
    opstelling en zijn deze ronde niet uitgevoerd; het mechanisme is wel in code geverifieerd.
